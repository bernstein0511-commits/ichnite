# Ichnite 解説原稿

質問された時にすぐ開けるよう、話す順番のまま書いてあります。丸暗記ではなく、
各見出しの内容を自分の言葉で話せるようにしておく想定です。
「Q. 」から始まる部分は想定質問と回答例です。

---

## 1. 一言で言うと

Ichnite（イクナイト）は、**Webページを読んでいて気になった単語やフレーズにマーカーを引き、
AIの解説付きで記録・復習できるChrome拡張機能**です。

英単語帳をイメージしてもらうとわかりやすいです。紙の単語帳と違うのは、
「ページを読みながらその場でマークするだけで、意味・類似語・対義語・例文をAIが自動生成し、
後からいつでも復習できる」という点です。

---

## 2. 課題・作った理由

読書中に知らない単語や気になる表現に出会っても、多くの場合はそのまま読み進めてしまい、
後で調べ直すことはほとんどありません。「気づいた瞬間」と「実際に調べて覚える瞬間」の間に
ギャップがあることが、語彙が身につかない大きな原因だと考えました。

Ichniteは、この「気づいた瞬間」にワンクリックで記録できるようにし、さらに
「記録して終わり」にならないよう、記録の可視化（連続記録日数・ヒートマップ等）で
継続的な復習につなげる設計にしています。

---

## 3. 全体構成（アーキテクチャ）

3層構成です。

```
Webページ（content script）
   │  ※ 直接localhostへ通信できないため
   ▼
background.js（拡張機能のService Worker）
   │  ※ ここがAPI通信を代行する
   ▼
FastAPI（バックエンド）
   │
   ▼
MySQL（Docker Compose）
```

- **拡張機能（フロントエンド）**: Manifest V3のChrome拡張機能。Chromiumベースの
  ブラウザ（Chrome / Edge / Brave）ならそのまま動作します。
- **バックエンド**: Python / FastAPI。REST APIでマーカー・ページ・AI解説・メモを管理。
- **DB**: MySQL。`docker compose up`だけでMySQLごと起動します。
- **AI**: OpenAI API（gpt-4o-mini）で解説・類似語・対義語・例文を生成。
  APIキーが無くても他の機能はすべて動きます（後述）。

Q. なぜバックエンドを自分で立てたのか？（Chrome拡張機能のstorage APIで十分では？）
A. マーカーを「どのページの、どの位置に、いつ記録したか」を横断的に集計・検索したり、
　 AIとの連携やページをまたいだ統計（ヒートマップ等）を出したりするには、
　 ローカルストレージより構造化されたDBの方が適していると判断しました。

## 4. デモの流れ（実演する順番）

実際に動かしながら説明する場合は、この順番で見せると機能が一通り伝わります。

### ① マーカーを引く
適当なWebページで文章の一部をドラッグ選択すると、選択箇所の近くに丸いカラーパレット
（黄・緑・青・赤・紫）がポップアップで出る。色をクリックするとその場でハイライトされ、
裏側でバックエンドにマーカーとして保存される。

同時にAI解説（意味・類似語・対義語・例文）が自動生成される（保存とは別で非同期に走るので、
マーカー自体はAI生成を待たずにすぐ引ける）。

### ② サイドパネルを開く
画面右下の☰ボタンでサイドパネルを開閉できる。ここには「このページのマーカー」または
「全ページのマーカー」の一覧が出て、それぞれの単語をクリックするとその位置までスクロールする
（別ページのマーカーならそのページへ遷移してからスクロールする）。

パネル上部のボタンで「ページ上のマーカー表示のON/OFF」「マーカーを引く時のカラー選択
ポップアップのON/OFF」も切り替えられる。

### ③ ハイライトにカーソルを合わせる
ページ上のハイライト済みの単語にマウスを乗せると、その場でメモの追加・編集・削除が
できる小さなポップアップが出る。

### ④ 記録帳ページを開く
サイドパネルの「記録帳」ボタン、または拡張機能アイコンから記録帳ページ（独立したタブ）
を開く。ここが管理画面にあたり、

- 全マーカーを表形式で一覧
- ページ／色／期間／キーワードで絞り込み検索
- 学習の可視化（下記⑤）
- 各マーカーの削除

ができる。

### ⑤ 学習の可視化（記録帳ページ上部）
GitHubのContributionグラフのような**ヒートマップ**で、直近18週間の記録の推移を表示。
月ラベル・曜日ラベル・凡例付きで、「直近◯日間で計◯件、記録した日は◯日」というサマリー文も出す。
あわせて「総マーカー数」「記録した日数」「現在の連続記録日数」「最長連続記録日数」を
カード表示し、色（黄緑青赤紫）ごとの内訳もバーで見せている。

### ⑥ 単語の詳細ページ
記録帳の各行から「表示」を押すと、その単語専用の詳細ページ（`marker_detail.html?id=...`）
に遷移する。メモ・AI解説・使用例・類似語・対義語・記録元ページをまとめて確認でき、
その場でメモの編集や「AI解説の生成／再生成」ができる。

### ⑦ 再読み込みしても消えない
ページをリロードしても、保存済みのマーカーは同じ位置に正しく復元される
（詳細は6章「出現順インデックス」参照）。

## 5. 機能一覧（早見用）

- テキスト選択→5色のカラーパレットでマーカーを作成
- 同じ箇所への二重登録を防止
- AI解説（意味・類似語・対義語・例文）の自動生成／手動での生成・再生成
- サイドパネル：一覧表示・絞り込み・メモ編集・削除・ページ内ジャンプ
- ページ上ハイライトへのホバーでメモ編集・削除
- マーカー表示ON/OFF、カラー選択ポップアップON/OFFの切り替え
- マーカー記録帳ページ：検索・絞り込み・学習統計（ヒートマップ／連続記録日数／色内訳）
- 単語ごとの詳細ページ
- 複数タブ間のリアルタイム同期（あるタブでの追加・削除・メモ編集が他タブにも反映）
- Chromium系ブラウザ（Chrome / Edge / Brave）で共通動作
- OpenAI APIキー未設定でもコア機能は完全動作（AI解説だけ無効化）

## 6. 技術的な工夫ポイント（質問されやすい箇所）

### 6-1. 同じ単語が複数回出てくる場合の位置特定（出現順インデックス方式）

**課題**：ページ内に同じ単語が複数回登場する場合、「何文字目か」のような単純な位置情報では
保存できない（DOM構造が変わりうるため）し、単純な文字列検索では常に最初の出現位置に
マーカーがついてしまう。

**解決**：選択した文字列が「ページ内で何番目に出現した同じ文字列か」（0始まりのインデックス）
だけをDBに保存する。復元時は同じロジックでページを走査し、同じインデックス番目の出現箇所を
探してハイライトを再構築する（`modules/textLocator.js`）。DBスキーマ上は`position_start`
`position_end`という列名だが、実体はこのインデックス値を転用している。

### 6-2. Private Network Access（PNA）とバックグラウンド経由のAPI呼び出し

**課題**：ページ上で動くcontent scriptから直接`http://localhost:8000`へfetchすると、
Chromiumの Private Network Access という仕組みによってブロックされ、開発時には
特殊なフラグ付きでブラウザを起動する必要があった（審査員に配布する上で大きな障害になる）。

**解決**：content script側は直接fetchせず、`chrome.runtime.sendMessage`で
拡張機能のバックグラウンド（Service Worker）にAPI呼び出しを依頼し、実際のfetchは
バックグラウンド側で行う構成に変更した。拡張機能自身の通信はPNAの制約を受けないため、
特殊な起動オプションなしで動作するようになった。

### 6-3. Shadow DOMによるUIの隔離

**課題**：サイドパネルやツールバーなどのUIをそのままページに挿入すると、
閲覧先のWebページのCSS（`* { margin: 0 }`のような強いリセットなど）の影響を受けて
見た目が崩れる可能性がある。

**解決**：拡張機能のUI要素はすべてShadow DOM（`modules/shadowHost.js`）の中に
入れている。`:host { all: initial }`でページ側から継承されるスタイルをリセットし、
UI専用のCSS（`content/panel-ui.css`）だけを読み込むことで、双方向にスタイルの影響を
遮断している。実際に引いたハイライト自体（`.ichnite-highlight`）はページ本文の一部として
存在する必要があるため、こちらは意図的にShadow DOMの外（通常のページCSS）に置いている。

### 6-4. AIキー未設定でも壊れない設計

OpenAI APIキーが無い、または通信エラーになっても、バックエンド全体が落ちたり
マーカーの保存自体が失敗したりしないよう設計している。AI解説の生成だけが独立して
失敗し、拡張機能側はそのエラーを無視してマーカー機能を継続する
（`services/ai_service.py`でキー未設定時は早期に分かりやすいエラーを返すのみ）。

### 6-5. 複数タブ間のリアルタイム同期

`chrome.runtime.sendMessage`は拡張機能ページ（記録帳など）には届くが、
他タブのcontent scriptには届かないという制約がある。そのため、変更通知は
一度バックグラウンドを経由し、開いている全タブへ`chrome.tabs.sendMessage`で
中継する構成にしている（`background.js`）。これにより、あるタブでマーカーを
削除すると、そのページを別タブで開いていてもハイライトが即座に消える。

## 7. データベース構成（4テーブル）

```
Page (閲覧ページ) 1 ── N Marker (ハイライトした語句)
                           │ 1
                 ┌─────────┴─────────┐
                 │ 1                 │ 1
           AiNote (AI解説)      MarkerBook (ユーザーのメモ)
```

- `pages`：マーカーを付けたWebページ（URL単位、重複登録は防止）
- `markers`：ハイライトした語句そのもの（色・出現順インデックス等）
- `ai_notes`：AIが生成した解説・類似語・対義語・例文（マーカー1件につき最大1件）
- `marker_book`：ユーザーが書いたメモ（マーカー1件につき最大1件）

一覧・検索・統計に使う`GET /marker_book/full`は、この4テーブルを結合して
1件ずつフラットな形で返すエンドポイント。

## 8. 想定質疑応答

**Q. なぜChrome拡張機能という形にしたのか？Webアプリではダメだったのか？**
A. 「読んでいる最中のページに直接マークできる」ことが本質的な価値なので、
　 任意のWebページ上で動作できる拡張機能の形が必須でした。Webアプリ単体では
　 他サイトのページ内容を直接操作できません。

**Q. マーカーの位置がページの内容変更（記事の更新など）で復元できなくなることは？**
A. あります。出現順インデックス方式は「ページの構造・文言が変わっていない」ことが前提です。
　 大きな更新があったページでは復元に失敗する場合がありますが、その場合はエラーとして
　 ログに出るだけで、他のマーカーの復元やページの表示には影響しません。

**Q. 複数人で使うとマーカーが混ざってしまうのでは？**
A. 現状はユーザー認証機構が無く、同じバックエンドを使う人同士でマーカーが共有される
　 設計になっています。個人利用や検証用途を想定した実装で、複数ユーザー対応は今後の課題です。

**Q. AI解説の精度・コストは？**
A. OpenAIのgpt-4o-miniを使用し、意味・類似語・対義語・例文の4項目をJSON形式で
　 生成するようプロンプトを設計しています（`services/prompt.py`）。1回あたりのコストは
　 小さいモデルなので低く抑えられますが、APIキーはユーザー自身のものを使う想定です。

**Q. セキュリティ面で気をつけたことは？**
A. ページ側の任意の文字列（選択テキストやメモ）をそのままHTMLに差し込むと
　 XSSにつながるため、表示前に必ずエスケープ処理を行っています。また、
　 `.env`にAPIキー等の秘密情報を置き、Gitの追跡対象から除外しています。

**Q. テストは書いているか？**
A. （正直に現状を答える：自動テストは未整備。手動での動作確認で品質を担保している旨と、
　 今後の課題として言及するのが誠実です。）

## 10. セットアップ手順（聞かれたら即答できるように）

1. `git clone` → `cd ichnite`
2. `backend/api/.env.example` を `.env` にコピー（DB設定は初期値のままでOK、
   AIを使うなら`OPENAI_API_KEY`だけ入力）
3. `docker compose up --build` でバックエンド＋MySQLを起動
4. Chromium系ブラウザの拡張機能ページで `extension` フォルダを読み込む
   （特殊な起動オプションは一切不要）

詳細は `README.md` の「アーキテクチャ概要」にファイル単位の役割一覧がある。
